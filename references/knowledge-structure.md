# Knowledge Base Structure

## Paths

**Personal KB Root:** `./knowledge/` (current working directory + /knowledge)

**Project KB Root:** `./docs/knowledge/` (current working directory + /docs/knowledge)

## KB Location Detection

Skills automatically detect which KB to use by checking convention paths for `.index/summaries.json`:

1. Check `./docs/knowledge/`
2. Check `./knowledge/`
3. Check `./`

**If multiple found:**

- Show list with document counts
- Prompt user to choose

**If none found:**

- Prompt: "Create KB at [1] ./docs/knowledge/ or [2] ./knowledge/? [1-2]:"
- Prompt: "Is this a project KB (fixed structure) or personal KB? [project/personal]:"
- Save `kb_type` in metadata.json

**KB Type determines:**

- **project**: Uses fixed categories (analysis, domains, adrs), rebuild is blocked
- **personal**: Uses dynamic categories, rebuild is allowed

**KB Types:**

- **Project KB**: Fixed 3-category structure (analysis, domains, adrs) for project context
- **Personal KB**: Dynamic categories, can be reorganized by rebuild

**Directory Layout (Personal KB):**

```text
./knowledge/
  .index/
    summaries.json    # Document metadata and search index
    metadata.json     # Aggregate statistics (kb_type: "personal")
    clusters.json     # Generated by rebuild
    clusters/         # Cluster details
  <category>/
    <document>.md     # Markdown documents with frontmatter
```

**Example (Personal KB):**

```text
./knowledge/
  .index/
    summaries.json
    metadata.json
  web-development/
    react-hooks.md
    nextjs-routing.md
  research/
    ai-paper-2024.md
```

**Example (Project KB):**

```text
./docs/knowledge/
  .index/
    summaries.json
    metadata.json (kb_type: "project")
  analysis/
    tech-stack.md
    architecture-diagram.md
    component-structure.md
  domains/
    api-patterns.md
    auth-flow.md
    order-management.md
  adrs/
    001-choosing-nextjs.md
    002-database-selection.md
```

**Project KB Categories (Fixed Structure):**

- **analysis/** - Tech stack, architecture, components, code structure
- **domains/** - Tech domains (api, auth, db) + Business domains (orders, users, payments)
- **adrs/** - Architectural Decision Records

## Document Format

Each document is a markdown file with YAML frontmatter:

```markdown
---
title: Document Title
tags: [tag1, tag2, tag3]
summary: Brief summary of content
key_concepts: [concept1, concept2]
related_topics: [topic1, topic2]
category: category-name
---

# Document content in markdown...
```

## Category Organization

**Project KB (Fixed):**

- Must use one of: `analysis`, `domains`, `adrs`
- During ingestion, AI suggests the most appropriate category from these three
- User can override but should stay within these categories

**Personal KB (Dynamic):**

- Lowercase with hyphens (e.g., `web-development`, `machine-learning`)
- Suggested by AI during ingestion
- User-editable
- Can be reorganized by rebuild into clusters

## Filename Convention

Filenames are:

- Sanitized from document title
- Lowercase, spaces replaced with hyphens
- Special characters removed
- Extension: `.md`

Examples:

- "React Hooks Guide" → `react-hooks-guide.md`
- "AI & Machine Learning" → `ai-machine-learning.md`
